# GalTransl 前端代码风格规范

> 适用范围：`desktop/`（Tauri v2 + SolidJS + TypeScript + Vite）全部源码。
> 本规范与仓库内已落地的工程化配置（`eslint.config.js`、` .prettierrc.json`、` .editorconfig`）一一对应，是它们的**人话说明 + 补充约定**。配置是强制基准，本文是约定补充。

---

## 0. 强制工具链（已安装，必须跑）

| 工具 | 版本 | 作用 |
| --- | --- | --- |
| ESLint | `^9`（flat config） | 静态检查：禁止 `any`、未用变量、Solid 反模式 |
| Prettier | `^3` | 格式化：引号/缩进/分号/行尾统一 |
| EditorConfig | — | 编辑器级：utf-8 / lf / 去尾空格 / 2 空格 |
| TypeScript | `^5.6`（`strict: true`） | 类型层纪律 |

提交前务必在 `desktop/` 执行：

```bash
npm run lint        # ESLint 检查，0 error 才能合
npm run lint:fix    # 自动修复可修复项
npm run format      # Prettier 格式化 src
npm run build       # vite 构建，确保类型与打包通过
```

CI / 本地自检以 `npm run lint` + `npm run build` 通过为准。

## 1. 格式化基线（由 Prettier + EditorConfig 保证）

- **缩进**：2 个空格，禁止 tab。
- **引号**：**双引号** `"..."`（`singleQuote: false`）。
- **分号**：**加**（`semi: true`）。
- **行宽**：`printWidth: 100`，超过自动换行。
- **尾逗号**：`trailingComma: "all"`，多行结构与函数参数一律保留尾逗号。
- **括号间距**：`{ foo: 1 }`（保留内部空格）。
- **箭头函数**：始终带括号 `(x) => x`（`arrowParens: "always"`）。
- **行尾**：`lf`（Unix 换行），`utf-8`，文件末尾保留一个空行，去除行尾空白。
- **不要手改格式**：任何排版争议交给 Prettier，`lint:fix` + `format` 一把梭，不要人工微调空格。

## 2. 命名约定

| 对象 | 规则 | 示例 |
| --- | --- | --- |
| 文件 / 目录 | `kebab-case`（短横线小写） | `review-page.tsx`、`undo-store.ts` |
| 组件文件 | 组件名 + `.tsx`，与导出组件同名（Pascal） | `ReviewPage.tsx` 导出 `ReviewPage` |
| 普通模块 | `kebab-case.ts` | `errors.ts`、`log-store.ts` |
| 组件（函数） | `PascalCase` | `function ReviewPage() {}` |
| 函数 / 变量 | `camelCase` | `loadProject`、`entryCount` |
| 常量 / 枚举值 | `UPPER_SNAKE_CASE` | `MAX_BATCH_SIZE = 100` |
| 类型 / 接口 | `PascalCase`，接口不加 `I` 前缀 | `CacheEntry`、`ProjectConfig` |
| CSS 类名 | **BEM**：`block__element--modifier` | `field__hint`、`btn--primary` |

- 组件、类型、常量用名词或名词短语；事件处理函数用 `handleXxx` / `onXxx`。
- 布尔变量用 `isXxx` / `hasXxx` / `canXxx`。
- 避免缩写歧义（`cfg` 可接受，`tmp2` 不行）。

## 3. 导入顺序（ESLint 推荐 + 人工约定）

统一顺序（各组之间空一行）：

1. 外部依赖（npm 包）：`solid-js`、`@tauri-apps/api` …
2. 项目内绝对/别名导入：`~/lib/...`、`~/stores/...`
3. 相对导入：`./`、`../`
4. 类型导入：`import type { ... }`（与值导入分开，便于 tree-shaking）

示例：

```ts
import { createSignal, For, Show } from "solid-js";
import { invoke } from "@tauri-apps/api/core";

import type { CacheEntry } from "~/lib/api/types";
import { getErrorMessage } from "~/lib/errors";

import { ReviewRow } from "./ReviewRow";
```

- 禁止 `export default`：组件一律**命名导出** `export function Xxx()`，便于重构与按需引入。
- 类型专用 `import type`，不要混在值导入里。

## 4. 组件范式（SolidJS）

- 一律**函数组件**，用 signals / stores 管理状态，不写 class 组件。
- 复杂 props 必须显式定义 `Props` 接口并标注：

  ```tsx
  interface ReviewRowProps {
    entry: CacheEntry;
    onSkip: (id: string) => void;
  }

  export function ReviewRow(props: ReviewRowProps) {
    return <div class="review-row">{props.entry.name}</div>;
  }
  ```

- 访问 props 用 `props.xxx`（响应式读取），**不要在组件体内解构 props 顶层**（会丢失响应性）。
- 列表渲染用 `<For>` / `<Index>`；条件渲染用 `<Show>` / `<Switch>`。
- 不要在 JSX 里写复杂逻辑，提取成函数或派生 `createMemo`。
- 事件处理器命名：`onClick` / `onInput` 等 Solid 原生属性（不是 React 的 `onClick` 也兼容，但保持统一用小写原生名）。

## 5. 类型纪律（**最高优先级**）

`tsconfig` 已开 `strict`，且 ESLint 规则 **`@typescript-eslint/no-explicit-any: "error"`** —— **任何 `any`（` : any` / `as any` / `as unknown as any`）都会让 lint 直接失败**。历史上 `any` 逃逸 51 处已全部收敛。

### 5.1 替代 `any` 的推荐手段

| 场景 | 做法 |
| --- | --- |
| 后端返回结构不确定的 JSON | 定义 `type ConfigValue = string \| number \| boolean \| null \| ConfigValue[] \| { [k: string]: ConfigValue }`，用 `Record<string, ConfigValue>` 做游标逐级下钻 |
| 动态键值对象 | `Record<string, T>`，不用 `{ [k: string]: any }` |
| 联合类型需要取字段 | 先用**类型守卫**或 `in` 收窄，再访问 |
| 第三方库无类型 | 写 `declare module` 或局部 `type X = {/* 已知字段 */}`，**不要** `as any` |
| 全局扩展（如 `window` 自定义属性） | 在 `global.d.ts` 里 `declare`：`interface Window { __addRecentProject?: (p: string) => void }` |
| 快照 / 撤销栈的异构数据 | 用联合类型，例如 `CacheEntry \| Record<string, unknown>` |

### 5.2 catch 块处理

`catch` 必须带绑定变量并收窄为错误：

```ts
import { getErrorMessage } from "~/lib/errors";

try {
  await invoke("do_thing");
} catch (e) {
  const msg = getErrorMessage(e); // 返回 string，统一处理
  toast.error(msg);
}
```

`src/lib/errors.ts` 已提供 `getErrorMessage(error: unknown): string`：是 `Error` 取 `.message`，否则 `String(error)`。**禁止** `err.message || String(err)`、`(err as any).code` 这类写法。
空 catch（确认可忽略，如目录已存在）必须加注释：`catch { /* 目录已存在则忽略 */ }`。

### 5.3 `unknown` 优于 `any`

外部输入（网络、invoke 返回值、JSON.parse）类型未知时，先用 `unknown`，再用类型守卫收窄。**绝不**用 `any` 跳过检查。

## 6. CSS / 样式约定

- 一律使用设计令牌，**禁止裸 hex / 裸颜色值**：

  ```css
  /* ✅ */
  .btn--primary { background: var(--color-primary); }
  /* ❌ */
  .btn--primary { background: #0066cc; }
  ```

- 类名遵循 **BEM**：`block__element--modifier`。
- 颜色、间距、圆角、字号等统一走 `var(--token)`，令牌定义在全局 `tokens.css`（不要新建零散 CSS 变量）。
- 组件级样式就近放，全局通用样式放 `src/styles/` 或 `tokens.css`，不要散落。
- 不用 `!important` 强行覆盖；优先提高选择器特异性或调整令牌。

### 文本选择（易踩坑）

- **禁止在 `html, body` 全局写 `user-select: none`**。本项目是翻译工具，复制原文/译文是核心工作流，全局禁选会导致整页左键拖选失效。
- 正确做法：选择能力默认开启（`body { user-select: text }`），**仅对 UI 外壳**显式关闭：
  - 标题栏 `.titlebar`（它有 `-webkit-app-region: drag` 拖窗，必须 `user-select: none`，否则会和窗口拖拽冲突）；
  - 活动栏 `.activitybar`、按钮类（`.btn` / `.entry-btn` 等，按钮 UA 默认即不可选，无需重复）；
  - 其他需要拖拽或点击的 chrome 容器。
- 编辑框 `input, textarea { user-select: text }` 防御性保留，确保输入/编辑字段始终可选中。

## 7. 注释与可读性

- 用中文注释（与项目现有风格一致），解释 **why** 而非 **what**。
- 复杂业务逻辑（字典合并、撤销栈、prompt 拼接）在关键分支写一句为什么这么做。
- 不要逐行翻译代码；不要写「TODO 个人情绪」类无意义注释。
- 公共函数 / 组件在签名上方写一行用途说明（可选，复杂时必写）。

## 8. 目录结构约定（保持现状，新增遵循）

```
desktop/
├─ src/
│  ├─ components/      # 通用 UI 组件（SidebarPanel、ActivityBar、TitleBar…）
│  ├─ pages/           # 页面级组件，按功能分子目录
│  │  ├─ home/  review/  settings/  dictionary/  project-config/
│  │  ├─ backends/  prompts/  translate/  wizard/
│  ├─ lib/             # 纯逻辑 / 接口层（api、errors、utils）
│  ├─ stores/          # 全局状态（undoStore、logStore、appStore）
│  ├─ styles/          # 全局样式 / 令牌
│  └─ global.d.ts      # 全局类型声明（window 扩展等）
├─ src-tauri/          # Rust 侧（不在本规范范围）
├─ eslint.config.js    # ESLint 扁平配置
├─ .prettierrc.json    # Prettier 配置
├─ .editorconfig       # 编辑器级格式
└─ package.json
```

- `lib/` 只放无副作用的纯函数 / 接口定义，不要塞 UI。
- `stores/` 放跨页面共享状态；页面内局部状态用组件内 `createSignal`，不入全局 store。
- 新增页面：在 `pages/` 下建子目录，组件命名 `XxxPage.tsx`，路由在 `App.tsx` 或路由配置处登记。

## 9. 易用性补充约定（面向零基础用户）

本项目用户为零基础汉化/翻译者，代码层面需配合以下 UX 约定：

- **按钮必须有可见文字，不能只靠悬停**：图标按钮必须配可见中文文字（图标 + 文字并列，如 `展开` / `跳过` / `删除`），不允许"纯图标 + `title` 悬停才懂"。左侧 ActivityBar 也采用「图标 + 文字标签」常显。`title` 仅作冗余提示，不得作为唯一含义来源。
- **配置字段用中文标签，不直接暴露英文参数名**：`ProjectConfigPage` 字段标签取自后端 `config-schema` 的中文注释（如 "项目级并行文件数"），原始英文 key（如 `common.workersPerProject`）与可选值（如 `[1-32]`）降为小字提示，不抢主体。无注释的兜底字段才人话化 key。
- **技术字段/JSON 配中文说明**：后端配置等 JSON 编辑器上方需配中文结构说明（如 `endpoint`=接口地址、`modelName`=模型名、`apiKey`=密钥），不让用户面对裸 key。
- **表单引导**：复杂表单字段配 `.field__hint` 说明文案 + 页面级 `.page-description` 总览，沿用现有模式。
- 错误反馈走右下角 Toast，不把错误信息塞进页面正文。

## 10. 提交规范

- 提交信息格式：**`type: 中文简述 (#issue)`**
  - `type` 取 `feat` / `fix` / `refactor` / `docs` / `style` / `test` / `chore`。
  - 示例：`fix: 收敛 ProjectConfigPage 的 any 逃逸 (#123)`
- **禁止**在提交信息中添加 AI 署名尾注（如 `Co-Authored-By: Claude <...>`）。
- 一个提交只做一类改动；规范/配置变更与业务改动分开提交。
- 提交前确保 `npm run lint` 与 `npm run build` 通过。
