# 前端代码风格统一性与零基础易用性审计

> 审计时间：2026-07-20
> 审计范围：`desktop/src`（SolidJS + Tauri 前端）

## 一、前端代码风格是否统一？

**结论：类型层面统一，但代码风格未做工程化强制统一——靠习惯，不靠工具。**

| 维度 | 状态 | 证据 |
|------|------|------|
| 代码格式化 / Lint 工具 | ❌ 缺失 | 无 ESLint、Prettier、EditorConfig 配置文件 |
| TypeScript 严格模式 | ✅ 强制 | `tsconfig.json`：`strict`、`noUnusedLocals`、`noUnusedParameters` 均为 true |
| `any` 类型逃逸 | ⚠️ 漏洞 | 全量 `: any` / `as any` / `as unknown as` 共 **51 处**，绕过 strict |
| CSS 设计令牌化 | ✅ 一致 | 845 处 `var(--token)` vs 仅 12 处裸 hex；统一走 `tokens.css` |
| CSS 命名 | ✅ 一致 | BEM 风格（`.btn` / `.field__hint` / `.wizard-panel`） |
| 组件导出风格 | ✅ 一致 | 42 个 `export function` + 2 个 `export const`，无 `export default` 混用 |
| 调试残留 | ✅ 干净 | 0 处 `console.log` / `debugger` |
| 组件范式 | ✅ 一致 | 均为 SolidJS 函数组件 + `Props` interface |

**小结**：新手接手或长期维护时风格会漂移；`any` 是主要的一致性漏洞（类型安全被局部绕过）。

## 二、零基础易用性 + 按钮提示，是否实现？

**结论：表单字段有引导，但按钮几乎没有提示；整体偏"技术向"，对零基础不够友好。**

### 已实现的引导（✅）
- **表单字段提示**：`.field__hint` 在字段下方显示灰色说明（如"用于识别与解析源文件格式"）。
- **页面级说明**：`.page-description` 在页顶给一句话概述。
- 向导 / 设置这类以表单为主的页面，基本有字段级引导。

### 未实现的引导（❌）
- **按钮提示几乎缺失**：
  - 全前端 **49 个 `<button>`**，仅约 **2 个** 带 `title` 属性（`grep <button[^>]*title=` = 2）。
  - 连同 div/span 充当按钮的 ~114 个可点控件，全量 `title=` 仅 **13 处（约 11%）**，且多集中在少数页面（review / dictionary / translate）。
  - **无自研 Tooltip 组件**（grep `Tooltip` = 0），仅用浏览器原生 `title=`——需悬停约 1 秒才出现、且仅悬停可见，不是"按钮附近常驻可见文案"，对零基础用户几乎等于没有。
- **技术化暴露，对零基础不友好**：
  - 项目设置页直接暴露 `config.yaml` 的原始 key（`backendSpecific`、`tokenStrategy`、`workersPerProject` 等 18 项），对象数组只显示"N 项（对象数组…）"。
  - 后端配置页需理解 `tokenStrategy` / `checkAvailable` / 各后端名（如 `OpenAI-Compatible`）。
  - 向导步骤名（"常用设置""翻译后端选择"）偏内部术语。

## 三、改进建议（若要做零基础友好版）

1. **新增轻量 Tooltip 组件**（CSS hover + 常驻小问号图标，不依赖原生 `title`），给所有主操作按钮（开始/停止/替换/保存/删除/新增/打开）配一句人话说明。
2. **按钮旁配可见文字**，而非只靠悬停（零基础用户不会主动悬停）。
3. **技术字段加"人话"标签**：如 `backendSpecific` → "翻译引擎密钥配置"，把 18 字段分组 + 每组图例。
4. **建立代码规范**：加 `.editorconfig` + ESLint(Prettier)，收敛 `any`，补一份风格约定文档。
