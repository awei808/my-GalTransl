# 完全重建前端，重建前端审美、设计、使用体验
- 舍弃react，采用Tauri + Rust + SolidJS
- 可保留文件：接口层文件，纯逻辑文件，svg图标绘制文件
- 状态管理使用createStore
- 统一右下角弹窗负责处理信息反馈
- 操作体验应该要类似vscode，但用不着vscode的太多功能
- 基本页面：
    1. 顶部留一排按钮（文件、帮助、重做、撤回等），点击出现下拉菜单，项目级功能接口放在这里
    2. 页面左侧留一列按钮（翻译控制台、校对审核、查找替换、问题检测、设置、字典、构建输出等），是处理翻译项目的常用界面切换按钮；点击可激活侧边栏或切换主界面
    3. 剩余部分为主界面，依托状态转换来切换页面
    4. 整体风格：蓝色（浅色 #0066CC；深色 #0099FF）为主色调，浅色填充色 #F0F5FF，深色填充色 #12161C；禁用emoji，采用svg绘制简洁清爽的图标，可参考vscode的图标；采用圆角矩形，圆角半径使用参考8px网格系统
    5. 主界面类似vscode的代码编写界面，可自由调整长宽，侧边栏也可自由调整宽度
    6. 主界面主要承接翻译接口界面，也可切换为设置界面，校对审核界面，翻译字典界面；默认为首页（只显示logo、制作人、项目地址、精简帮助信息）
    7. 翻译控制台界面只有项目被打开时，才会出现；为纯信息面板 + 启动流程，无文件浏览器，无条目列表，所有报错信息通过右下角 Toast 通知。

       顶部一个长宽比4:1的圆角矩形（矩形长度略小于主页面长度），矩形内左 2/3 显示：
         - 当前项目的总行数
         - 翻译进度（百分比 + 进度条）
         - 流程完成情况（使用步骤指示器展示 7 阶段状态，匹配 stage_index/stage_total）
         - 翻译速度（条/秒）
         - ETA（预计剩余时间）
       右 1/3 显示一个启动流程按钮（带下拉菜单选择翻译后端），
         - 下拉菜单展示可用的翻译引擎（来自 fetchTranslators()）
         - 点击按钮 → 提交翻译任务（submitJob） → Toast 反馈开始/失败

       顶部矩形下方分为两栏，可自由调整两栏的宽度：
         - 左栏：当前提示词显示（只读纯文本，来自 latest_prompt_preview）
         - 右栏：译文拼接显示（只读纯文本，来自 latest_assembled_preview）

       轮询策略：2~3s 间隔调用 fetchProjectRuntime() 刷新所有只读数据。
       错误条目数 / 失败条目数：暂不显示（等前端 demo 出来后再补）。

    8. 设置界面通过点击页面左侧设置按钮的方式，切换主界面实现，包括系统设置和翻译项目设置，简单可视化；读取后端系统和翻译项目配置文件，显示参数名称，将参数对应的注释作为简要解释（可选），层级按照正常方法处理

    9. 校对审核界面只有打开项目文件时，才会出现；通过点击页面左侧审核校对按钮，再打开侧边栏文件浏览器选择文件的方式，切换主界面实现。

       条目展示方式：
         - 每条 JSON 默认显示 3 行：问题标记行、原文行（pre_src）、译文行（pre_dst）
         - 3 行文字右侧显示按钮：展开字段、跳过检查、删除该 JSON
         - 点击文本即可出现光标，直接编辑（无需编辑框包裹）
         - 只读辅助信息直接用 text 显示
         - 译文字段用 <input> 包裹

       展开字段后：
         - 显示 CacheEntry 的全部 18 个字段，按正常 JSON 文本数据排列
         - 只有译文字段（pre_dst / proofread_dst）可编辑，其余只读

       编辑与保存：
         - 自动保存：编辑失焦后自动触发 saveCacheFile
         - 依赖全局重做/撤回机制（ctrl+z / ctrl+y）

       导航：
         - 页面内滚动条上下滑动浏览条目
         - 支持跳转至特定 index 的 JSON（输入编号跳转功能）
         - 增删改快捷按钮：只实现逻辑层接口和数据结构，暂不页面展示（为未来功能预留）

    10. 翻译字典界面，通过点击页面左侧字典按钮，再打开侧边栏对应字典（人名替换表字典/gpt字典/译前替换字典/译后替换字典）的方式，切换主界面实现；翻译字典界面实现纯文本和表格卡片两种显示方式，可编辑

    11. 构建输出没有单独界面，点击按钮后检测缓存文件完整性，若全部文件完整则构建全部文件输出，若部分文件不完整则调用确认弹窗对话询问用户是否仅构建完整文件输出，若全部文件不完整则不构建

查找替换功能：
  - 位于左侧按钮栏，点击后激活侧边栏的查找替换面板，不切换主界面
  - 功能对标 VSCode 侧边栏的 Find in Files：搜索框 + 替换框 + 切换大小写/全词匹配
  - 搜索结果按文件分组展示，显示行号，点击结果可跳转到校对审核页的对应条目
  - 替换无需预览，直接执行，通过右下角 Toast 通知结果（替换了多少条）
  - 替换操作依赖重做/撤回机制（ctrl+z 可撤销替换）
  - 查找替换侧边栏可与校对审核主界面同时显示（侧边栏 + 主界面的典型布局）

问题检测面板：
  - 位于左侧按钮栏，点击后激活侧边栏的问题列表面板，不切换主界面
  - 按文件分组展示问题列表，显示问题类型、条目 index、问题描述
  - 点击问题行 → 跳转到校对审核页的对应条目
  - 问题列表可筛选（按类型、按文件）
- 键盘快捷操作：至少保留ctrl+z/x/c/v/f/h/a/s，与vscode的对应功能一致；其中ctrl+f/h打开的是右上角的文件内查找替换功能，而不是左侧按钮的文件夹内全文件的查找替换功能
- 保留全局重做/撤回机制（ctrl+z 撤销，ctrl+y 重做），校对审核页面的自动保存依赖此机制实现
- 要有一个确认弹窗对话类，用于在缺少非关键数据时向用户确认是否执行操作，弹窗内部可嵌入html
- 条目列表按文件加载，非全项目加载；一个文件最大不超过10000行（通常为10到500行），超过10000行则启用虚拟滚动，最多同时加载1000行
- 各页面的局部状态由各页面自己维护，不入appStore全局状态
